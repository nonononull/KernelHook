# Strategy Matrix Goldens / 策略矩阵黄金文件

Source-of-truth goldens for per-kernel strategy behavior. Never hand-edit
`values/*.yaml` — regenerate via
`scripts/test.sh strategy-matrix --accept <avd-or-serial>`
after reviewing the DRIFT output.

每个内核版本在本目录留下一份黄金快照。`values/*.yaml` 由工具生成，不要手改。
通过 `scripts/test.sh strategy-matrix --accept <设备名>` 更新，更新前先看 DRIFT 输出。

## Files

- `expectations.yaml` — human-written rules (expectation type per capability,
  value ranges, enum rules). Modify when adding a new capability or when the
  rule for an existing one changes.
- `values/<device-id>.yaml` — machine-generated per-device snapshots.
  Diff via `scripts/test.sh strategy-matrix <device>`.
- `survival.tsv` — Y/N matrix across devices, one row per (capability, strategy).
  Regenerated by `--accept`.
- `consistency_runs.log` — append-only history of consistency_check results.

## Adding a new AVD / device

1. Boot the target with `kernelhook.ko kh_consistency_check=1` loaded.
2. `scripts/test.sh strategy-matrix --dump <name>` — produces candidate yaml.
3. Review: does it match `expectations.yaml` rules? If yes:
4. `scripts/test.sh strategy-matrix --accept <name>` — writes golden and updates
   `survival.tsv`.
5. Commit.

## Adding a new capability

1. Define strategies in `src/strategies/`.
2. Add entry to `expectations.yaml` with correct expectation type.
3. Re-run `--accept` on each known-good device to refresh goldens.

## Expectation types

| Type | Meaning |
|---|---|
| `scalar_all_strategies_equal` | All successful strategies must return same value |
| `enum_matches_rule` | Value must satisfy the `rule` expression |
| `scalar_per_kernel_build` | Same `linux_banner_sha256` → same value |
| `probed_may_vary` | Value must be in `range` / `allowed`; strategies may differ |
| `function_pointer_any_valid` | Function pointer in kernel text range; strategies may return different impls |
| `procedural_only` | Success = side effect achieved; no value comparison |

## Drift review checklist

When `strategy-matrix <device>` reports DRIFT, decide per capability:
- Expected drift (new kernel, new build, intentional API rename) → `--accept`.
- Unexpected drift (regression, corrupted loader probe, walker heuristic
  catching wrong pattern) → investigate before accepting.
