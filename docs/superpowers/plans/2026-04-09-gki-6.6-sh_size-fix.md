# GKI 6.6 / 6.12 `sh_size` Fix Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `kh_test.ko` load on Pixel_35 (GKI 6.6) and Pixel_37 (GKI 6.12) while preserving PASS on Pixel_33 (5.15) and Pixel_34 (6.1), by shrinking `.gnu.linkonce.this_module` `sh_size` in `kmod_loader` to match the running kernel's `sizeof(struct module)`.

**Architecture:** Two coordinated changes inside `tools/kmod_loader/`:

**Tech Stack:** C (freestanding-compatible subset), POSIX file I/O, native host compiler for tests (`cc`), Android NDK clang for cross-compiles, Android emulator AVDs for integration verification.

- [ ] **Step 4: Create `tools/kmod_loader/patch_this_module.h`**

```c
/* SPDX-License-Identifier: GPL-2.0-or-later */
#ifndef KH_PATCH_THIS_MODULE_H
#define KH_PATCH_THIS_MODULE_H

#include <stdint.h>
#include <stdio.h>

/* kver_preset mirrors the struct in tools/kmod_loader/kmod_loader.c:47.
 * Defined here so both kmod_loader.c and the unit test link without
 * pulling the full loader. Field order/types MUST stay in sync with
 * that definition — kmod_loader.c should include this header and
 * REMOVE its local definition to guarantee a single source of truth. */
struct kver_preset {
    int      major;
    int      minor;
    uint32_t mod_size;
    uint32_t init_off;
    uint32_t exit_off;
};

/* Shrink the sh_size of a .gnu.linkonce.this_module section header
 * (pointed to by sh_size_ptr) to match the running kernel's
 * sizeof(struct module), taken from a resolved preset.
 *
 * The signature takes a raw uint64_t* rather than an Elf64_Shdr*
 * deliberately — it avoids a dependency on <elf.h>, which is missing
 * on macOS hosts. The caller (patch_module_layout in kmod_loader.c)
 * passes &this_mod->sh_size, which is a uint64_t field (Elf64_Xword).
 *
 * Required by Android 15 GKI 6.6+ downstream kernels, which enforce
 * that the .gnu.linkonce.this_module section size exactly equals
 * sizeof(struct module) at load time and reject otherwise with
 * -ENOEXEC. Safe on older kernels because we never zero the section
 * and the kernel loader on pre-6.6 does not check the size.
 *
 * Defensive: if either init_off or exit_off would land outside the
 * shrunk range, we refuse and leave sh_size untouched (the kernel
 * will then still reject, surfacing a clear error).
 *
 * Return:
 *   1  shrink applied — caller may log a trace line
 *   0  no-op (preset->mod_size is zero, or current sh_size already <= target)
 *  -1  refused — a relocation target would be cut off by the shrink
 */
static inline int
maybe_shrink_this_module_sh_size(uint64_t *sh_size_ptr,
                                 const struct kver_preset *preset)
{
    if (!sh_size_ptr || !preset) return 0;
    if (preset->mod_size == 0) return 0;
    if (*sh_size_ptr <= preset->mod_size) return 0;

    uint32_t rela_max = preset->init_off;
    if (preset->exit_off > rela_max) rela_max = preset->exit_off;

    /* Reloc writes a pointer (8 bytes on ARM64) at r_offset. */
    if ((uint64_t)rela_max + sizeof(void *) > preset->mod_size) {
        fprintf(stderr,
            "kmod_loader: WARN: refusing sh_size shrink 0x%llx -> 0x%x "
            "(reloc target 0x%x would be cut off)\n",
            (unsigned long long)*sh_size_ptr,
            preset->mod_size, rela_max);
        return -1;
    }

    fprintf(stderr,
        "kmod_loader: shrink this_module sh_size 0x%llx -> 0x%x\n",
        (unsigned long long)*sh_size_ptr, preset->mod_size);
    *sh_size_ptr = preset->mod_size;
    return 1;
}

#endif /* KH_PATCH_THIS_MODULE_H */
```

## Task 3: Vendor `.ko` walker for `sizeof(struct module)`

**Purpose:** Provide a preset-free fallback source for `VAL_THIS_MODULE_SIZE` so `kmod_loader` works on new GKI kernels that haven't been entered into `kmod/devices/*.conf` yet. Reuses the existing cached vendor-`.ko` scan logic already at `kmod_loader.c:326` (`crc_from_vendor_ko`).

**Files:**
- Modify: `tools/kmod_loader/kmod_loader.c`

- [ ] **Step 1: Read the existing `crc_from_vendor_ko()` and understand its caching**

Open `tools/kmod_loader/kmod_loader.c` and read lines ~326–410 (the full `crc_from_vendor_ko` function, including the static `ko_buf` / `ko_loaded` cache). Notice:
- It walks `/vendor_dlkm/lib/modules`, `/vendor/lib/modules`, `/system/lib/modules`, `/odm/lib/modules`, `/lib/modules`
- On first call, it caches the first readable `.ko`'s `__versions` entries into `ko_crcs[]`
- It frees `ko_buf` after extracting CRCs — the ELF buffer is NOT retained

To also extract `.gnu.linkonce.this_module` sh_size, we need to either (a) add a second static `ko_this_module_size` variable populated during the same scan, or (b) do a second pass. Option (a) is cleaner and reuses the same file-open cost.

## Notes for the executing engineer

- **Do not skip Task 1.** The TDD cycle there is the only pure-function verification of the shrink helper. AVD tests cannot distinguish "shrink helper is wrong" from "shrink helper was skipped".
- **Do not combine Tasks 2 and 3/4.** They are independent commits with independent rollback paths — if Task 3/4 break something, we want to be able to revert just those commits.
- **Task 5 is not optional.** Shipping without the Pixel_34 run is shipping without the empirical evidence that justified choosing "always shrink" over "6.6+ only shrink".
- **If the helper's defensive reloc-cutoff check ever fires in real life**, that's a sign a new kernel's init/exit offsets have grown past `sizeof(struct module)` — which would indicate the preset itself is stale and the `kmod/devices/*.conf` entry needs to be updated. Treat it as a diagnostic, not a bug in the helper.
