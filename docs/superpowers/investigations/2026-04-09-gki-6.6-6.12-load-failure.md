# Investigation: kh_test.ko Load Failure on GKI 6.6 / 6.12 (Pixel_35 / Pixel_36 / Pixel_37)

- **Date:** 2026-04-09
- **Investigator session:** KernelHook main worktree (`/Users/bmax/workspace/github/KernelHook-investigation`, off `main@4037ea6`)
- **Scope:** read-only investigation; no changes to `tools/`, `kmod/`, `tests/`, `scripts/test_avd_kmod.sh`, or `.github/workflows/`. Plan 3 (`feat/kbuild-interop`) work runs in parallel and owns those paths.
- **Parent branch:** `main @ 4037ea6` (after Plan 2 Milestone E T17, before Plan 3 Task 1).

> **中文摘要**：Pixel_35/36/37 (GKI 6.6/6.12) 上 `kh_test.ko` 无法加载的根因是
> **Android 15 GKI downstream kernel 新增了严格的 `.gnu.linkonce.this_module` section size 校验** —
> 要求 ELF 中该 section 的 `sh_size` 精确等于 running kernel 的 `sizeof(struct module)`。我们的 kh_test.ko 被编译成 `sh_size=0x800`（padding），但 6.6 kernel 期待 `0x600`、6.12 kernel 期待 `0x640`，mismatch 直接被拒绝为 `-ENOEXEC`。仅调整 ELF 中的 `sh_size` 即可加载成功并通过 25/25 tests，根因定位完成。

### 1.1 Definitive kernel-side rejection message

Captured via `emulator -show-kernel` serial console on both Pixel_35 and Pixel_37:

```
module kh_test: .gnu.linkonce.this_module section size must match
                the kernel's built struct module size at run time
```

This is **an Android GKI downstream kernel patch** (not upstream Linux as of 6.12). Upstream Linux does not enforce this check; Android 15 GKI does. It fires from the module loader’s `elf_validity_check()`/`load_module()` path and returns `-ENOEXEC` to userspace.

### 1.2 ELF section header on built module

```
$ llvm-readelf -S tests/kmod/kh_test.ko | grep this_module
[11] .gnu.linkonce.this_module PROGBITS … 005840 000800 00  WA  0   0 64
```

`sh_size = 0x800` (from `-DTHIS_MODULE_SIZE=0x800` padding in `tests/kmod/Makefile`, added so `kmod_loader` could *shrink* at runtime). The padding value itself is intentional — the problem is that `kmod_loader` no longer shrinks (see §2).

### 1.3 Kernel configs (Pixel_35, `/proc/config.gz`)

Relevant excerpts:

```
CONFIG_MODULES=y
CONFIG_MODULE_UNLOAD=y
CONFIG_MODVERSIONS=y
CONFIG_MODULE_SIG=y
CONFIG_MODULE_SIG_PROTECT=y        ← Android downstream
# CONFIG_MODULE_SIG_FORCE is not set
CONFIG_CFI_CLANG=y
CONFIG_ARM64_BTI=y
CONFIG_SHADOW_CALL_STACK=y
CONFIG_ARCH_SUPPORTS_CFI_CLANG=y
CONFIG_RANDSTRUCT_NONE=y
```

`MODULE_SIG_FORCE` is *not* set, so unsigned modules can load (tainted). `MODULE_SIG_PROTECT` is the Android-specific scheme that enforces signatures against a restricted keyring but still allows unsigned modules in root-adb mode; it is *not* the cause here.

### 1.6 Side-observation (non-blocking)

After the shrink fix, the kernel still prints (non-fatally):

```
kh_test: disagrees about version of symbol module_layout
```

The module is marked `T` (tainted) but **does load** and **all tests pass**. Reason: `MODULE_SIG_FORCE` is unset, so modversions mismatch is a warning, not ENOEXEC. Our CLI `--crc module_layout=0x4e276f37` was extracted from `kernel-ranchu` via `scripts/extract_avd_crcs.py`; on Pixel_37 the real CRC is `0x797f2b3e`. Why the running kernel disagrees with its own image CRC is a second-order question (possibly a kallsyms vs ksymtab lookup discrepancy on kCFI-enabled kernels) — unrelated to the ENOEXEC root cause and can be tracked separately.

## 2. Hypotheses Tested

### H1. Resolver never matched the 6.6 / 6.12 device preset, so `init_off/exit_off/mod_size` stayed at 0 and `patch_module_layout()` was skipped.

**Status:** REJECTED.

- `tools/kmod_loader/devices_table.generated.c` (lines 115–158) contains `gki_6_12_android_api37` and `gki_6_6_android15` entries with correct `match_kernelrelease` prefixes (`"6.12."`, `"6.6."`).
- The reason no "init offset" line appeared in the loader log is that `patch_module_layout()` at `tools/kmod_loader/kmod_loader.c:956-968` logs only when `old_off != preset->init_off`. The freestanding build already hard-codes `0x188/0x5b8` for 6.6 and `0x188/0x5f8` for 6.12 via `tests/kmod/Makefile:79-83`, which exactly match the conf-file presets, so the log message is silent.
- Not a resolver bug; relocation offsets were already correct at build time.

### H2. `module_layout` symbol was removed in kernel 6.4+ (the commit that refactored `struct module` into `mod_mem[]`).

**Status:** REJECTED.

- `/proc/kallsyms` on Pixel_35 (6.6.30): `ffffffc080173e0c T module_layout` and `ffffffc08166f760 r __ksymtab_module_layout`. The symbol still exists on Android 15 GKI.

### H3. `CONFIG_MODULE_SIG_FORCE` / `MODULE_SIG_PROTECT` rejects the unsigned module with `-ENOEXEC`.

**Status:** REJECTED.

- `CONFIG_MODULE_SIG_FORCE is not set` in `/proc/config.gz`. `MODULE_SIG_PROTECT` does not reject unsigned modules in this adb-root environment.
- The fix (sh_size shrink) bypasses any signature check entirely, and the same signing configuration still allows the shrunk module to load. If signing were the gate, shrinking would not help.

### H4. kCFI hash algorithm or storage changed in kernel 6.6 so the `from virtio_input.ko` borrow is invalid, causing `__cfi_slowpath` to ENOEXEC at load.

**Status:** REJECTED.

- The `Exec format error` originates from *ELF validity checking*, which runs **before** any kCFI call-site validation. kCFI violations on a loaded module would manifest as a runtime `BRK #0xc47x` panic, not `-ENOEXEC` from `init_module()`.
- After the shrink fix (with the same borrowed kCFI hashes) all 25 test cases including direct function calls into `init_module` pass. If the borrowed hashes were wrong, the very first `init_module()` invocation would panic.

### H5. New `MODULE_VERSION_MAGIC` prefix (randstruct, gcc_version, arm64_bti_c) not in our vermagic string.

**Status:** REJECTED.

- `CONFIG_RANDSTRUCT_NONE=y` → no randstruct prefix.
- The loader already patches vermagic via `probe_procfs:/proc/version` at runtime (see trace `avail=153`), producing the exact string the running kernel expects. Vermagic is not the rejection reason.
- Confirmed: after sh_size shrink, the module loads with unchanged vermagic logic.

### H6. Android GKI 6.6/6.12 added a strict `sizeof(struct module)` check against `.gnu.linkonce.this_module` `sh_size`.

**Status:** **VERIFIED — root cause.**

- Kernel printk (captured verbatim from `-show-kernel` serial console on both Pixel_35 and Pixel_37):
  ```
  module kh_test: .gnu.linkonce.this_module section size must
                  match the kernel's built struct module size at run time
  ```
- Build artifact sh_size: `0x800` (padding from `-DTHIS_MODULE_SIZE=0x800`).
- Expected by 6.6 kernel: `0x600`; by 6.12 kernel: `0x640` (both match our `kmod/devices/*.conf` presets).
- Minimal intervention (hex-edit `sh_size` in the Shdr entry to the preset value, no other changes) makes both kernels load the module and complete 25/25 tests.
- Intervention script: `scripts/investigations/2026-04-09-gki-6.6/shrink_this_module.py` (included below as inline reference; see Appendix A).

## 3. Root Cause

**Android 15 GKI 6.6 and API 37 GKI 6.12 kernels enforce an exact match between the ELF `.gnu.linkonce.this_module` section size and the kernel’s `sizeof(struct module)`.** This check does not exist on upstream Linux or Android 14 GKI 6.1 or earlier AVD kernels, which is why `kh_test.ko` with its `0x800`-padded section loads on Pixel_30…34 but is rejected with `-ENOEXEC` on Pixel_35/36/37.

KernelHook's `tools/kmod_loader/kmod_loader.c` used to shrink `this_mod->sh_size` in the ELF based on the resolved `preset->mod_size`, but this code was removed in Plan 2 Milestone E T17 (commit `4037ea6` and its parents) after a bisect found that a combination of **(zeroing the entire section) + (shrinking sh_size)** broke exporter module init on Pixel_34 GKI 6.1. The safer fix at that point kept only the `.rela` offset patching. That T17 fix is correct for 6.1 but leaves the pre-condition of an exact sh_size unsatisfied on 6.6+, which the new GKI check now enforces.

Direction and magnitude of the fix: **restore sh_size shrinking in `patch_module_layout()`**, but **without** the zeroing that originally broke 6.1.

## 4. Fix Directions (graded from simplest to cleanest)

### Direction A — 1-line restore of sh_size shrink (MINIMAL)

In `tools/kmod_loader/kmod_loader.c:patch_module_layout()`, after the `.rela` offset patching, add:

```c
if (preset->mod_size && this_mod->sh_size > preset->mod_size) {
    fprintf(stderr, "kmod_loader: shrink this_module sh_size 0x%llx -> 0x%x\n",
            (unsigned long long)this_mod->sh_size, preset->mod_size);
    this_mod->sh_size = preset->mod_size;
}
```

- **Preconditions already satisfied:**
  - `resolve(VAL_THIS_MODULE_SIZE, …)` is already called in `kmod_loader.c:1810` and feeds `resolved.mod_size`.
  - All reloc `r_offset`s for init/exit (6.6: `0x188/0x5b8`, 6.12: `0x188/0x5f8`) are strictly less than the shrunk size.
  - The Plan 2 T17 bisect showed the **zeroing** (not the shrink alone) broke 6.1 — the zeroing is already gone; pure sh_size shrinking should therefore be safe on 6.1.
- **Work:** ≈ 5 lines + regression test on Pixel_33/34 (6.1 must still pass).
- **Risk:** Low. Only changes an ELF header field, not payload bytes.
- **Caveats:** Must verify on Pixel_33/34/30 that the shrink alone (no zeroing) does not reintroduce the T17 regression. Expectation: passes.
- **Dependency on Plan 3:** None. This patch touches `tools/kmod_loader/kmod_loader.c` which is in Plan 3's scope, so it **must** land either (a) on `feat/kbuild-interop` itself, or (b) after Plan 3 merges back to main, to avoid conflicts.

### Direction B — Also lower `THIS_MODULE_SIZE` padding, or make `tests/kmod/Makefile` emit the exact preset size at build time

- Build-time preset allows the shrink to be a no-op at runtime on the matching kernel, but the loader still needs Direction A to handle physical-device / unverified-AVD kernels whose struct size we don't know at build time.
- **Work:** ≈ 10 lines in `tests/kmod/Makefile`.
- **Risk:** Low, but duplicates the preset database.
- Not necessary if Direction A lands. Listed for completeness.

### Direction C — Runtime probe of `sizeof(struct module)` from a loaded vendor .ko

Rather than trust the conf-file preset, walk a loaded `.ko`’s `.gnu.linkonce.this_module` section size via `/proc/modules` + `/sys/module/<name>/sections/.gnu.linkonce.this_module` and read back. This removes the per-kernel preset maintenance burden.

- **Work:** ≈ 40 lines (new strategy in `resolver.c`, add to `VAL_THIS_MODULE_SIZE` chain above `config_automatch`).
- **Risk:** Medium. `/sys/module/<m>/sections/...` may be restricted in late-boot Android, and reading another module's bytes cross-references the loaded-module ELF format.
- Highest long-term value; can be paired with Direction A as a fallback.

### Direction D — Probe `sizeof(struct module)` from the kernel image on disk (host-side extract)

Add `scripts/extract_avd_struct_module_size.py` similar to the existing CRC extractor, and teach `scripts/test_avd_kmod.sh` to pass `--this-module-size=0xNNN` as a CLI override. Fixes the test pipeline without touching `kmod_loader` at all.

- **Work:** ≈ 30 lines of Python + 1 line in `scripts/test_avd_kmod.sh`.
- **Risk:** Very low — only affects test harness.
- Doesn't help real-device Ring-3 usage, but unblocks CI immediately.

**Recommended:** Direction A (minimal kernel-loader restore) + Direction D (CI quickfix) in parallel. Direction C is the long-term polish.

## 6. Secondary Issue (Out-of-Scope Follow-Up)

On Pixel_37 (6.12), `extract_avd_crcs.py` reads `module_layout=0x797f2b3e` from the kernel image, but the running kernel then emits `kh_test: disagrees about version of symbol module_layout`. The module still loads (tainted) because `MODULE_SIG_FORCE` is off. This is a second-order bug — CRC extraction logic may be reading a stale or wrong CRC location on kCFI-protected 6.12 kernels. It is **not** related to the ENOEXEC root cause. File as separate ticket. Suggested probe: dump `__crc_module_layout` section directly from the running kernel's `/proc/kallsyms` address range vs the value `extract_avd_crcs.py` reads from the kernel-ranchu image.

## Appendix A — Hypothesis-test hex-edit script

```python
# scripts/investigations/2026-04-09-gki-6.6/shrink_this_module.py
# Usage: python3 shrink_this_module.py <kh_test.ko> <new_size_hex>
# Patches .gnu.linkonce.this_module (section index 11 in kh_test.ko)
# sh_size to the given value. For 6.6 kernels use 0x600; for 6.12 use 0x640.
import struct, sys
path, new_size = sys.argv[1], int(sys.argv[2], 0)
with open(path, "rb") as f: b = bytearray(f.read())
e_shoff = struct.unpack_from("<Q", b, 0x28)[0]
e_shentsize = struct.unpack_from("<H", b, 0x3a)[0]
sh = e_shoff + 11 * e_shentsize  # section 11 = .gnu.linkonce.this_module
old = struct.unpack_from("<Q", b, sh + 0x20)[0]
print(f"old sh_size=0x{old:x}  new sh_size=0x{new_size:x}")
struct.pack_into("<Q", b, sh + 0x20, new_size)
open(path, "wb").write(b)
```

This is a minimal hypothesis-test tool, not a production fix. The production fix belongs in `tools/kmod_loader/kmod_loader.c:patch_module_layout()` per Direction A above.
