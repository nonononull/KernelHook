# Design: Fix kh_test.ko Load Failure on GKI 6.6 / 6.12 via `.gnu.linkonce.this_module` sh_size Shrink

- **Date:** 2026-04-09
- **Target branch:** `fix/gki-6.6-sh_size-shrink` off `main @ 4037ea6`
- **Related investigation:** `docs/superpowers/investigations/2026-04-09-gki-6.6-6.12-load-failure.md`
- **Status:** Design approved (pending spec review)

## 1. Problem

Android 15 GKI 6.6+ downstream kernels enforce that the ELF `.gnu.linkonce.this_module` section size exactly equals the running kernel's `sizeof(struct module)`. `kh_test.ko` is built with `-DTHIS_MODULE_SIZE=0x800` padding so its section is 0x800, while the running kernels expect 0x600 (6.6) and 0x640 (6.12). Load fails with `-ENOEXEC` and the kernel printk:

```
module kh_test: .gnu.linkonce.this_module section size must match
                the kernel's built struct module size at run time
```

Pre-existing infrastructure (`VAL_THIS_MODULE_SIZE` value in the resolver, `this_module_size` field in `kmod/devices/*.conf`) already carries the expected values but `patch_module_layout()` never applies them — Plan 2 Milestone E T17 removed the sh_size shrink after a bisect found that **zeroing the section + shrinking** broke exporter init on Pixel_34 (6.1). Single-step shrinking (without zeroing) was never isolated by that bisect.

## 2. Goals and Non-Goals

**Goals**
- `kh_test.ko` loads and passes 25/25 tests on Pixel_35 (GKI 6.6) and Pixel_37 (GKI 6.12).
- Existing PASS status preserved on Pixel_33 (5.15) and Pixel_34 (6.1).
- Preset-less fallback: if a new kernel isn't in `kmod/devices/` yet, `kmod_loader` can still discover `sizeof(struct module)` from a vendor `.ko` on device.
- No change to `kh_crc` (Contract 4).
- Fix is minimal (~60 LoC) so it rebases cleanly against Plan 3 (`feat/kbuild-interop`) whichever merges first.

**Non-Goals**
- Fixing the secondary "disagrees about version of symbol module_layout" taint warning on 6.12 — tracked separately.
- Touching `tests/kmod/Makefile` padding value (stays 0x800; loader shrinks at runtime).
- Changes to `scripts/test_avd_kmod.sh` or CI matrix (Direction D dropped — root-cause fix makes it unnecessary).

## 3. Approach (Direction A + Direction C combined)

Two coordinated changes, both within `tools/kmod_loader/`:

**A. `patch_module_layout()` shrink step.** After the existing `.rela.gnu.linkonce.this_module` r_offset patching, unconditionally shrink `this_mod->sh_size` to `preset->mod_size` whenever the resolved size is non-zero and strictly less than the current section size. No zeroing of payload bytes. No conditional version gating — applies on all kernels uniformly.

**C. Runtime probe from vendor `.ko` on disk.** Extend `strategy_probe_ondisk_module` so that `VAL_THIS_MODULE_SIZE` queries walk `/vendor/lib/modules/*.ko` (reusing the existing walker used by `crc_from_vendor_ko`), parse each candidate's ELF section table, and return the `sh_size` of its `.gnu.linkonce.this_module` section. Since vendor `.ko` files are built from the same GKI kernel tree as the running kernel, their `.gnu.linkonce.this_module` sh_size is authoritative — equal to `sizeof(struct module)` the running kernel enforces.

**Why both, not just A.** Direction A alone closes the bug on any kernel already in `kmod/devices/*.conf`. Direction C removes the ongoing maintenance burden of keeping the preset database in sync with every new GKI release — a new kernel plugged into a fresh AVD just works if it has any vendor module installed (all GKI devices do).

## 4. Component Breakdown

### 4.1 `tools/kmod_loader/kmod_loader.c`

Two changes:

(a) Add file-scope helper:
```c
/* Probe sizeof(struct module) from a vendor .ko on disk.
 * Reads the sh_size of the .gnu.linkonce.this_module ELF section
 * of any parseable /vendor/lib/modules/*.ko. Vendor .ko files are
 * built by the same GKI kernel that's running, so their
 * .gnu.linkonce.this_module sh_size equals the running kernel's
 * sizeof(struct module). Returns 0 on success, -1 if no parseable
 * .ko or no such section found. */
int sizeof_struct_module_from_vendor_ko(uint32_t *out);
```
Implementation reuses the directory-walking scaffold already in `crc_from_vendor_ko()` (factor into a small internal `for_each_vendor_ko()` iterator if convenient; otherwise copy the walker pattern — the function is short).

(b) In `patch_module_layout()`, after the existing relocation-offset loop (currently ending at `tools/kmod_loader/kmod_loader.c:971`), add:

```c
/* Direction A: shrink this_module sh_size to match running kernel's
 * sizeof(struct module). Required by Android 15 GKI 6.6+ downstream
 * strict size check (see docs/superpowers/investigations/
 * 2026-04-09-gki-6.6-6.12-load-failure.md). Safe on older kernels:
 * shrink-only, no zeroing — Plan 2 T17 bisect evidence implicates
 * the zeroing step as the real root cause of the 6.1 regression,
 * and this fix never zeros. Guard against reloc targets being cut
 * off by the shrink (defensive assertion — all known presets have
 * init/exit < mod_size, but future kernels could break this). */
if (preset->mod_size && this_mod->sh_size > preset->mod_size) {
    uint32_t rela_max = 0;
    if (preset->init_off > rela_max) rela_max = preset->init_off;
    if (preset->exit_off > rela_max) rela_max = preset->exit_off;
    if (rela_max + sizeof(void *) <= preset->mod_size) {
        fprintf(stderr,
            "kmod_loader: shrink this_module sh_size 0x%llx -> 0x%x\n",
            (unsigned long long)this_mod->sh_size, preset->mod_size);
        this_mod->sh_size = preset->mod_size;
    } else {
        fprintf(stderr,
            "kmod_loader: WARN: refusing sh_size shrink 0x%llx -> 0x%x "
            "(reloc target 0x%x would be cut off)\n",
            (unsigned long long)this_mod->sh_size,
            preset->mod_size, rela_max);
    }
}
```

No other changes in this file.

### 4.2 `tools/kmod_loader/strategies/probe_ondisk.c`

Replace the `VAL_THIS_MODULE_SIZE` no-op fallthrough with:

```c
if (id == VAL_THIS_MODULE_SIZE) {
    uint32_t sz;
    if (sizeof_struct_module_from_vendor_ko(&sz) == 0) {
        out.available = 1;
        out.u64_val = sz;
        strncpy(out.source_label, "probe_ondisk:vendor_ko_this_module",
                sizeof(out.source_label) - 1);
        return out;
    }
}
```

Inserted before the existing `/* struct_module offsets: punt to … */` comment. Declaration of `sizeof_struct_module_from_vendor_ko` added to top of file via extern or a shared header — choose whichever pattern the file currently uses for `crc_from_vendor_ko` (they're peers).

### 4.3 `tools/kmod_loader/tests/test_resolver.c`

Existing file already defines mock versions of all strategy functions. Two new cases:

1. **shrink_applies_when_resolved_smaller**: seed `resolved.mod_size = 0x600`, build a fake ELF with `.gnu.linkonce.this_module` sh_size = 0x800, call `patch_module_layout`, assert final sh_size == 0x600.
2. **shrink_noop_when_resolved_zero**: same fake ELF, `resolved.mod_size = 0`, assert sh_size unchanged.
3. **shrink_refused_when_reloc_would_be_cut**: seed `resolved.mod_size = 0x100`, `init_off = 0x200`, assert sh_size unchanged and warn message path exercised.

Host-side only; no cross-compile needed.

### 4.4 Files explicitly NOT touched

- `kmod/devices/*.conf` — `this_module_size` fields already correct; schema unchanged.
- `tests/kmod/Makefile` — `THIS_MODULE_SIZE=0x800` padding stays.
- `kmod/shim/shim.h` — owned by Plan 3 (`feat/kbuild-interop`); unchanged.
- `tools/kh_crc/**` — Contract 4 forbids byte-level changes; untouched.
- `scripts/test_avd_kmod.sh` — owned by Plan 3; unchanged.
- `.github/workflows/**` — owned by Plan 3; unchanged.

## 5. Data Flow

`VAL_THIS_MODULE_SIZE` resolver chain (unchanged order):

```
CLI (--this-module-size=…)
  └─▶ PO (probe_ondisk_module)       ← EXTENDED: now handles THIS_MODULE_SIZE
        └─▶ CE (config_explicit)
              └─▶ CA (config_automatch)   ← existing preset path from kmod/devices/*.conf
                    └─▶ CF (config_fuzzy)
                          └─▶ PD (probe_disasm, currently unimpl for this value)
                                └─▶ END
```

`patch_module_layout()` flow becomes:

```
1. find .gnu.linkonce.this_module and .rela.gnu.linkonce.this_module   (unchanged)
2. walk rela entries, patch r_offset for init_module/cleanup_module     (unchanged)
3. NEW: if preset->mod_size > 0 and sh_size > mod_size and rela targets
        are within the new size, shrink sh_size to mod_size
```

## 6. Error Handling

| Failure | Behavior |
|---|---|
| PO walker finds no vendor `.ko` or all parses fail | Return unavailable; resolver continues to CA/CF |
| All resolver strategies return unavailable | `resolved.mod_size == 0`; shrink branch guard-skipped; loader emits `WARN: sizeof(struct module) unknown; module may be rejected on strict kernels`; load continues (fix-less behavior) |
| `preset->mod_size` smaller than a reloc target (defensive) | Shrink refused; warn logged; loader continues with original sh_size (kernel will reject on 6.6+ — surfaced to user for diagnosis) |
| Kernel still rejects after shrink | Loader falls through existing `finit_module → init_module` path; user diagnoses via serial console — identical to current behavior for other rejection reasons |

### 7.4 Rollback plan

If Pixel_34 (6.1) regresses on fix branch:

1. **Do not revert Direction C** — PO extension is independent of shrink behavior and unrelated to any T17 risk.
2. **Downgrade Direction A to version-gated variant**:
   ```c
   int strict_size_check = (kmajor > 6 || (kmajor == 6 && kminor >= 6));
   if (strict_size_check && preset->mod_size &&
       this_mod->sh_size > preset->mod_size) { … }
   ```
3. Re-run AVD matrix; ship downgraded variant; file follow-up issue to bisect the 6.1 shrink regression more narrowly.

This fallback keeps 6.6/6.12 fixed at the cost of leaving 5.x/6.1 on the 0x800 padding path.

## Appendix A — Why not Direction D (test harness CLI override)

Direction D in the investigation report would add `--this-module-size=` to `scripts/test_avd_kmod.sh` as a workaround. Rejected here because:

1. It fixes only the CI pipeline; real-device Ring 3 deployments would still fail on 6.6+.
2. It violates the "root cause first" principle — the bug is in `kmod_loader`, not in the test script.
3. It would need to be reverted once Direction A lands anyway.

### Known Follow-Up: Ring 3 on GKI 6.6+ (out of scope for this fix branch)

**Symptom:** On Pixel_35/Pixel_37, after Ring 1 passes, the export_link_test phase tries to load `exporter.ko` and the kernel rejects it with the same printk our investigation identified: `module exporter: .gnu.linkonce.this_module section size must match the kernel's built struct module size at run time`.

**Why our fix doesn't catch it:** `exporter.ko` and `importer.ko` are built via `kmod/mk/kmod.mk` (not `tests/kmod/Makefile`), which uses `kmod/shim/shim.h:305`'s default `#define THIS_MODULE_SIZE 0x440`. Their `.gnu.linkonce.this_module` `sh_size` is therefore `0x440` (1088 bytes), which is **smaller** than what 6.6 (`0x600`) and 6.12 (`0x640`) require. Our `maybe_shrink_this_module_sh_size()` helper is shrink-only — its guard `*sh_size_ptr <= preset->mod_size` returns 0 when the section is already smaller than the kernel target, so no patching occurs and the kernel rejects the original `0x440`.

**Why it's not a regression:**
- On `main @ 4037ea6` (the fix branch's base), Pixel_35/Pixel_37 fail Ring 1 first. The runner kills the AVD before Ring 3 ever runs, so Ring 3 6.6+ failure was invisible.
- After Task 1+2, Ring 1 6.6+ passes, which **exposes** this pre-existing latent failure in the shim/kmod.mk build path.
- The shim path's 6.6+ failure has the same root cause (Android 15 GKI strict size check) but a different fix vector (build-time padding bump), and lives in `kmod/shim/shim.h` and/or `kmod/mk/kmod.mk` — both files in the parallel Plan 3 (`feat/kbuild-interop`) workstream's exclusive scope.

**Decision:** Document and defer. This fix branch ships Ring 1 fix only. The Ring 3 fix is a **separate, simpler change** to be tracked as a follow-up — see the handoff document `docs/superpowers/handoffs/2026-04-10-gki-6.6-sh_size-fix-handoff.md` for the full remaining work spec.
