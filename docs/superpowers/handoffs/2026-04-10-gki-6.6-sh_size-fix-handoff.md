## Remaining work item 1 — Ring 3 padding bump for GKI 6.6+

### Root cause (verified empirically this session)

- `exporter.ko` and `importer.ko` are built via `kmod/mk/kmod.mk`, which compiles with the default values from `kmod/shim/shim.h`.
- `kmod/shim/shim.h:305` defines `THIS_MODULE_SIZE = 0x440` (= 1088 bytes).
- The resulting `.ko` files have `.gnu.linkonce.this_module` `sh_size = 0x440`.
- On GKI 6.6 the kernel requires `sizeof(struct module) = 0x600` (1536 bytes); on GKI 6.12 it's `0x640` (1600 bytes).
- `0x440 < 0x600` → our shrink helper's guard `*sh_size_ptr <= preset->mod_size` returns 0 (no shrink needed from the helper's perspective), and the section stays at 0x440 → kernel rejects with the same `section size must match` printk.
- Verified ELF inspection: `llvm-readelf -S exporter.ko` → `[11] .gnu.linkonce.this_module PROGBITS … 0044c0 000440 …`

### Fix (recommended)

**Single-line change in `kmod/shim/shim.h`:**

```c
#ifndef THIS_MODULE_SIZE
-#define THIS_MODULE_SIZE 0x440
+#define THIS_MODULE_SIZE 0x800
#endif
```

This bumps the default padding for **all shim-based builds** (exporter.ko, importer.ko, any future kmod.mk-built module) to 0x800. The `kh_test.ko` build path is unaffected because `tests/kmod/Makefile` already overrides to `0x800`. After the bump:

- 5.4/5.10/5.15/6.1 kernels: shim has 0x800, kernel doesn't enforce size, helper shrinks to whatever preset says (0x400) — same code path that already works for kh_test.ko on those kernels
- 6.6 kernel: shim has 0x800, helper shrinks to 0x600
- 6.12 kernel: shim has 0x800, helper shrinks to 0x640

The helper added in Task 1 is already in place and will handle all four kernels uniformly.

### Risks

| Risk | Mitigation |
|---|---|
| Bumping `THIS_MODULE_SIZE` enlarges every shim-built `.ko` by `0x800 - 0x440 = 0x3c0` (960 bytes). For exporter.ko: 51992 → 52952 bytes (~1.8% bigger). | Negligible for any realistic use; not a constraint anywhere we care about. |
| Some `.ko` consumer hard-codes the old 0x440 size (unlikely) | Search `kmod/` and `tools/` for the literal `0x440` before changing — should only appear in `shim.h`. |
| Plan 3 may already have changed the default | Check `feat/kbuild-interop` for any modification to `kmod/shim/shim.h:305` before applying. If Plan 3 already bumped it, the work is done — just re-run the matrix to verify. |

## Remaining work item 5 — Independent follow-up: 6.12 `module_layout` CRC mismatch

During investigation (see `docs/superpowers/investigations/2026-04-09-gki-6.6-6.12-load-failure.md` §1.6), Pixel_37 (GKI 6.12) was observed to print `kh_test: disagrees about version of symbol module_layout` after our shrink fix lets the module load. The module still loads (tainted) because `MODULE_SIG_FORCE` is unset, so this is non-blocking for the current bug, but it indicates a real second-order issue:

- `scripts/extract_avd_crcs.py` reads `module_layout=0x797f2b3e` from the kernel-ranchu image on Pixel_37
- The running kernel rejects that CRC as wrong
- Possible causes: kCFI-protected 6.12 stores the CRC differently, the script reads from a stale section, or the running kernel computes the CRC differently than what's in the image.

This is **completely orthogonal** to the sh_size fix. File as its own issue and investigate independently. Suggested probe:
- Dump the actual running kernel's CRC for `module_layout` via a probe module that reads `__crc_module_layout` from `/proc/kallsyms` if exposed
- Compare against what `extract_avd_crcs.py` reports
- If they differ, the script's image-parsing logic on 6.12 needs an update
